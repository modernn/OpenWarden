# ADR-032: Child identity-key hardware binding — StrongBox P-256 device-binding key attests TEE-resident Curve25519 identity (resolves the StrongBox/Curve25519 incompatibility)

Status: Proposed
Date: 2026-06-22
Amends: **docs/PROTOCOL.md §7.2/§7.3, §10.7** (child keygen + parent verification + conformance), **docs/CRYPTO.md §1/§3/§10** (key inventory + keygen recipe + attestation hard-rules), **docs/adr/025-pairing-handshake-direction-attestation-sas.md D2 checks 3–4 + D7 + test plan** (the STRONGBOX-attestation requirement now binds the P-256 device-binding key), **docs/adr/029-tier2-attestation-posture.md D2 + Security-note + test plan** (the leaf-pubkey binding → `K_bind` + new check 4b apply on Tier 2 too; on Tier-2 TEE devices `K_bind` inherits ADR-029's StrongBox→TEE keygen fallback), **docs/DEFENSES.md #4 + Patterns A/E**, **docs/ANDROID_COMPAT.md §3**, **docs/MIGRATION.md + docs/PROVISIONING_V2.md** (stale "StrongBox-attested Ed25519/X25519" wording)
Relates: **ADR-025** (pairing handshake — direction, four-key SAS, the attestation trust model this preserves), **ADR-029** (Tier-2 TEE attestation posture — the TEE-residency residual disclosed here is consistent with that tier's accepted downgrade), **ADR-031** (the merged #21 SpkiBinding/SpkiAssertion signed by the child Ed25519 identity key, which this keeps intact), **ADR-015/019** (the one Ed25519+JCS signing rule, unchanged for the identity key), **ADR-001/023** (Pixel-class enforcement tier); docs/DEFENSES.md #4/#9/#14 + Patterns A/E (anti-device-swap), docs/ATTACKS.md H3 (pubkey substitution, CRITICAL); issue #22 (the child identity key this scopes); the ADR-025 D5 parent-side pairing issues that verify the chain

## Context

Issue #22 ("child StrongBox identity key") is the bedrock that fills the `IdentityKeyProvider` seam ADR-031 left returning `null`, unblocking #21's `SpkiBindingSigner`. Preparing to implement it surfaced a **BLOCKER-level conflict in already-ratified canon**, confirmed by a read-only crypto review:

**Android StrongBox cannot generate or attest Curve25519 keys.** The hardware-backed StrongBox (Pixel 7 Titan M2 / the StrongBox KeyMint contract) supports EC NIST curves (P-256/`secp256r1`, P-384, P-521), RSA, AES, HMAC, 3DES — **not** Ed25519 or X25519. `KEY_ALGORITHM_ED25519` / `KEY_ALGORITHM_XDH` arrived in AndroidKeyStore at API 33 but produce **TEE/software-backed** keys only; `setIsStrongBoxBacked(true)` on a Curve25519 spec throws `StrongBoxUnavailableException` (or yields a non-`STRONGBOX` `securityLevel`). There is **no** path to a `securityLevel == STRONGBOX` attestation record for an Ed25519 or X25519 key.

Yet canon, as ratified, demands exactly that. The assumption "Ed25519 + X25519 generated *in StrongBox*, attested at `securityLevel == STRONGBOX`" is load-bearing in at least eight places: PROTOCOL §7.2 step 2 / §7.3 check 4 / §10.7; CRYPTO §1 key inventory / §3 keygen recipe (whose code sample is also **internally contradictory** — it calls `KeyPairGenerator.getInstance(KEY_ALGORITHM_EC)` and initializes it with `ECGenParameterSpec("ed25519")`, which the EC generator does not accept) / §10 attestation hard-rules; ADR-025 D2 check 3 + D7 (mandatory `STRONGBOX` attestation); DEFENSES #4 + Patterns A/E; ANDROID_COMPAT §3. The §7.2 keygen step the child is told to run cannot execute on the declared Tier-1 hardware, and the **mandatory** ADR-025 D2.3/D7 STRONGBOX check would refuse the only key the spec tells the child to generate. **Implementing #22 verbatim produces a child that cannot pair.** No existing ADR resolves this; the red-team review (research/07) caught the TLS-provenance gap (TR1) but not this curve/hardware incompatibility.

**Why this is canon, not an implementer's call.** Resolving it changes ratified normative sentences in protected `proto`/crypto surface (the §7 handshake, the §7.2 wire schema, the attestation trust model). Per CLAUDE.md, crypto/protocol changes REQUIRE an ADR. This ADR records the resolution; the implementation (#22 child side, plus the ADR-025 D5 parent-side verification) follows it, attested-and-tested.

**Confirm-before-implement gate (knowledge-cutoff caveat).** The StrongBox/Curve25519 platform claim above is asserted from documentation, not bench-verified in this analysis. Before any #22 key-handling code lands, a maintainer MUST confirm it on a real Pixel 7: generate an `EC secp256r1` `setIsStrongBoxBacked(true)` key with an attestation challenge, retrieve the chain, parse the extension, and confirm `securityLevel == STRONGBOX (2)`; and separately confirm that the same flags on an `Ed25519`/`XDH` spec do **not** yield a `STRONGBOX` record (expect `StrongBoxUnavailableException` or a TEE-level result). If — contrary to current docs — a Pixel 7 *does* attest Curve25519 at StrongBox level, this ADR is unnecessary and should be withdrawn in favor of the original §7.2.

## Options

The security anchors any option must hold: **H3** (pubkey substitution, CRITICAL — ATTACKS) is defended by "the pinned child keys are hardware-bound + parent-pinned + recovery-gated rotation"; **DEFENSES #4/#14 + Pattern E** require a real hardware **attestation cert chain** proving genuine, unrooted, GREEN-boot, locked, allow-listed hardware, bound to the pinned key; **fail-closed** forbids silently degrading to a weaker posture without disclosure.

- **A. Make the identity key EC P-256 (change the protocol curve).** Then "pin against the StrongBox key" is literal and strongest. **Rejected — severe, wide blast radius:** it swaps the entire signature suite from Ed25519 to ECDSA-P-256 and the sealed box from X25519 to an ECDH-P-256 construction, detonating the libsodium `crypto_box_seal` envelope (PROTOCOL §6 / CRYPTO §4, byte-for-byte conformance §10.1), the JCS+Ed25519 signing invariant (ADR-015/019), the BIP39→Ed25519/X25519 parent-key derivation (CRYPTO §2), the four-key SAS HKDF (§7.4), **and the already-merged #21 `SpkiBinding`/`SpkiAssertion`** (Ed25519 via `net.i2p.crypto:eddsa`). It is a §8 format break → v2, forcing re-pair from scratch. Not worth it when the hybrid keeps real attestation without the curve swap.
- **B. Software Ed25519/X25519 wrapped under a StrongBox AES MasterKey; attestation dropped/downgraded.** Generate the identity keys in software, store privkeys in `EncryptedSharedPreferences` under a StrongBox-backed AES `MasterKey` ("StrongBox-wrapped, not StrongBox-generated"). **Rejected for v0.x:** there is no hardware attestation cert chain over the identity, so "parent confirms genuine, unrooted Pixel hardware" (DEFENSES #14 / Pattern E) is lost and device-swap is no longer caught at pairing. This *is* the SAS-only posture ADR-025 D7 explicitly forbids on the Pixel tier ("we never drop to SAS-only"). It is roughly the Tier-3 escape hatch ANDROID_COMPAT already describes — fine as a documented Tier-3 fallback, not the Tier-1 path. Disclosing the downgrade does not make it conform to a ratified ADR decision.
- **C. Hybrid — StrongBox EC P-256 *device-binding key* attests, and signs over, the TEE-resident Curve25519 identity (chosen).** Preserves real STRONGBOX attestation and H3/DEFENSES#14, keeps the identity curve Ed25519 (so #21 and the whole protocol surface are untouched), and has the smallest blast radius — at the cost of a **disclosed** "Curve25519 private key is TEE-resident, *bound to* StrongBox rather than *in* it" residual.

## Decision

Adopt **Option C**. Eight parts.

**D1 — Three child keys, two homes.** At pairing the child generates:
- **`K_bind` — device-binding key: EC P-256 (`secp256r1`), in StrongBox.** `setIsStrongBoxBacked(true)`, `setAttestationChallenge(provisioning_nonce)`, `setUnlockedDeviceRequired(true)`. This is the only hardware-attestable key; its attestation cert chain is what proves genuine hardware. Permanent per-device (retained for periodic re-attestation, DEFENSES #9 / CRYPTO §10).
- **`K_id` — identity key: Ed25519, TEE/AndroidKeyStore** (`KEY_ALGORITHM_ED25519`, `setUnlockedDeviceRequired(true)`). The long-term identity the parent pins; signs the `SpkiAssertion` (ADR-031) and any child-authored signed object. Unchanged curve ⇒ #21 intact.
- **`K_enc` — encryption key: X25519, TEE/AndroidKeyStore** (`KEY_ALGORITHM_XDH`). The sealed-box audience (PROTOCOL §6); the parent pins it.

`K_id`/`K_enc` private keys never leave the keystore; on a Pixel they are TEE-backed (StrongBox cannot hold them — D-context). They are **bound to** the StrongBox `K_bind` by D2, not resident in it.

**Tier-2 (committed Samsung/OnePlus, ADR-029).** `K_bind` is EC P-256 — which is **TEE-attestable as well as StrongBox-attestable** — so the hybrid maps onto Tier 2 unchanged: `K_bind` inherits ADR-029's keygen posture (attempt `setIsStrongBoxBacked(true)`, fall back to TEE on `StrongBoxUnavailableException`, **never** `SOFTWARE`), and its attestation `securityLevel` is accepted at `STRONGBOX` **or** `TRUSTED_ENVIRONMENT` per ADR-029 D1 (Tier 1 stays StrongBox-only). The binding signature (D2) and check 4b (D4) are **identical on both tiers** — only `K_bind`'s own residency tier differs, exactly as ADR-029 already accepts for the device key. This means ADR-029's "leaf-cert pubkey == `child_ed25519_pub`" assertions are superseded here too (see D4a).

**D2 — The binding statement: `K_bind` signs over the two Curve25519 public keys (freshness-bound).** The child constructs

```
ChildKeyBinding { v: 1, child_ed25519_pub, child_x25519_pub, provisioning_nonce }
```

and signs the **RFC 8785 JCS canonical bytes of the object minus the signature** with `K_bind` (ECDSA-P-256 / SHA-256), producing `child_binding_sig`. The `provisioning_nonce` inside the body ties the binding to *this* pair attempt (a captured binding from a prior attempt fails freshness). This is the cryptographic link from hardware-attested StrongBox to the pinned identity: an attacker who swaps `child_ed25519_pub` or `child_x25519_pub` must also forge a `K_bind` signature, which requires the StrongBox key it does not have.

**Domain separation (required clause, mirrors ADR-031 D2).** `child_binding_sig` is the **only** object signed by `K_bind` (ECDSA-P-256), and its JCS key-set `{v, child_ed25519_pub, child_x25519_pub, provisioning_nonce}` is **disjoint** from every other signed wire object — `SpkiAssertion` `{v, spki_sha256}` (signed by the child **Ed25519** key, ADR-031), and `SignedCommand`/`SignedBundle`/`SignedHeartbeat` (signed by the **parent** Ed25519 key, ADR-030). A signature is therefore non-transferable across objects on **two** independent axes: a different signing key (`K_bind` P-256 vs. child/parent Ed25519) **and** a different algorithm (ECDSA-P-256 vs. Ed25519), with disjoint canonical bodies on top. No cross-object replay exists.

**ECDSA malleability (explicit).** `child_binding_sig` is a **one-shot accept/reject** verified once at pairing; it is **never** used as a uniqueness key, replay token, or dedup key (the single-use `provisioning_nonce` + the SAS do that). ECDSA `(r, n−s)` malleability is therefore not exploitable — a malleated-but-valid signature still only re-proves the same fact. The verifier MUST NOT cache or compare raw `child_binding_sig` bytes for freshness, and MUST NOT depend on signature determinism (AndroidKeyStore/StrongBox does not guarantee RFC 6979).

**D3 — §7.2 child POST gains `child_binding_sig`; the attestation chain now attests `K_bind`.** The child's POST to the parent pairing endpoint becomes:

```json
{
  "v": 1,
  "child_ed25519_pub": "base64url(32)",
  "child_x25519_pub":  "base64url(32)",
  "child_attestation_cert_chain": ["base64(DER)", "base64(DER)", "base64(DER)"],
  "child_binding_sig": "hex(ECDSA-P-256 over JCS{v,child_ed25519_pub,child_x25519_pub,provisioning_nonce})"
}
```

`child_attestation_cert_chain` is the chain for **`K_bind`** (the leaf is the P-256 binding key's attestation cert). Adding `child_binding_sig` is a **`proto` wire-format change** ⇒ agent-blocked, human-implemented; the `proto/` schema and parent-kmp type land with the implementing issues, not here.

**D4 — Parent verification (amends §7.3 / ADR-025 D2 checks 3–4); pin only after every check passes; any failure refuses the pair (fail-closed).** The parent MUST verify, refusing on any failure:
1. cert chain parses and roots in the **Google Hardware Attestation root** (Tier-1; ADR-029 widens the accepted root set on Tier-2, unchanged here);
2. leaf attestation challenge == the `provisioning_nonce` it issued;
3. `verifiedBootState == VERIFIED` (GREEN), bootloader **locked**, device is an **allow-listed model**, attestation **`securityLevel == STRONGBOX`** — **now satisfiable, because the attested key is `K_bind` (P-256), not a Curve25519 key**;
4. leaf-cert public key == the presented **`K_bind`** public key (the leaf attests `K_bind`, not `child_ed25519_pub`);
4b. **`child_binding_sig` verifies** as a valid ECDSA-P-256 signature by the leaf (`K_bind`) public key over `JCS{v:1, child_ed25519_pub, child_x25519_pub, provisioning_nonce}` — this is the new check that binds the pinned Curve25519 keys to the attested hardware; failure refuses the pair.
Then per §7.4 (unchanged, four-key SAS) the parent compares the six emojis and, on **Match**, pins `(child_ed25519_pub, child_x25519_pub)`. No pin occurs unless attestation **and** the binding signature **and** the SAS all pass.

**D4a — The leaf-pubkey change applies on every tier; it amends ADR-029 and ADR-025's check-4 body + test plans, not only the §7.3 banner.** Checks 4 ("leaf == `K_bind`") and 4b (`child_binding_sig`) are **global to §7.3**, which both Tier 1 and Tier 2 inherit. Therefore every canon location that still binds the attested leaf to `child_ed25519_pub` is superseded: **ADR-025** D2 check 4 and its test-plan attestation bullet; **ADR-029** D2, its Security-note, and its test plan (`leaf-cert pubkey == child_ed25519_pub`). On **all** tiers the leaf attests `K_bind` and the Curve25519 keys are bound to it by `child_binding_sig`; ADR-029's "only two relaxations (root, security-level)" statement is widened by this one additional, tier-independent change. These ADRs carry an `ADR-032`-amended banner/note so an implementer following either cannot build a verifier that rejects an ADR-032-compliant child.

**D5 — The four-key SAS (§7.4 / ADR-025 D2a) is unchanged and still load-bearing.** It binds `parent_ed25519_pub ‖ parent_x25519_pub ‖ child_ed25519_pub ‖ child_x25519_pub`. `K_bind` is an attestation witness, not a pinned communication key, so it is **not** in the SAS — the binding signature (D2/D4 check 4b) is what ties `K_bind` to the pinned keys, and the SAS remains the human MITM catch over the four pinned keys. (A MITM cannot substitute a Curve25519 key without breaking either check 4b or the SAS.)

**D6 — Periodic re-attestation uses `K_bind`.** The 7-day re-attestation (DEFENSES #9 / CRYPTO §10 "every 7 days via refreshed attestation challenge") challenges **`K_bind`** with a fresh parent nonce; the pinned `K_id`/`K_enc` are unchanged by re-attestation. This is strictly cleaner than the old design (the previously-"attested" Ed25519 key was never attestable).

**D7 — Fail-closed + disclosed residual.** Any failure — `StrongBoxUnavailable` for `K_bind`, chain/nonce/boot/lock/model/level check fail, binding-sig fail, SAS mismatch — **refuses the pair** and surfaces the failure to the parent; nothing is left half-pinned (ADR-025 "Blocking" §). The **disclosed residual**: the Curve25519 *private* keys are TEE/keystore-resident, **bound to** StrongBox via D2, not StrongBox-resident. Against the declared threat model (ATTACKS §1 — kid with adb, no JTAG/physical extraction) TEE residency is already the accepted posture for Tier-2 (ADR-029 D1); here it is the Tier-1 residual created by the platform's curve limits, and it is disclosed to the parent at pairing exactly as ADR-029 discloses its TEE downgrade. Attestation stays **mandatory** (D4.3) and SAS-only stays out of scope — this does **not** drop to ADR-025 D7's forbidden SAS-only posture, because a real STRONGBOX attestation (of `K_bind`) is still required and still gates the pin.

**D8 — Blast radius is contained to provenance, not primitives.** The signature suite (Ed25519 + JCS, ADR-015/019), the sealed box (X25519, PROTOCOL §6), BIP39 derivation (CRYPTO §2), and the merged #21 `SpkiBinding`/`SpkiAssertion`/`Ed25519.kt` are **unchanged** — the identity stays Ed25519, only its *storage + attestation provenance* changes. `IdentityKeyProvider`'s interface (`identityPublicKey(): ByteArray?`, `sign(message): ByteArray?`) is unchanged; only its KDoc claim "signs inside StrongBox, key never leaves hardware" must be corrected at implementation time to "signs with the TEE-resident child identity key, attestation-bound to a StrongBox device-binding key; the private key never leaves the keystore" (recorded here so the security-critical comment is not silently false). This is **not** a `v` format bump for the existing log/bundle/event formats; only the §7.2 pairing POST gains a field (a forward-compatible §7 addition gated behind the new pairing issues).

## Consequences

**Good:**
- The BLOCKER is resolved: §7.2 keygen is now runnable on the declared hardware, and the mandatory STRONGBOX attestation (D4.3) is satisfiable because it attests `K_bind` (P-256).
- H3 / DEFENSES #4/#14 / Pattern E preserved: a real Google-rooted STRONGBOX chain still proves genuine, unrooted, GREEN, locked Pixel hardware; the binding signature chains the pinned Curve25519 identity to that hardware; the four-key SAS still catches MITM key substitution.
- #21 and the entire Ed25519/X25519 protocol surface are untouched (D8) — smallest blast radius; no v2 re-pair.
- One Ed25519+JCS signing rule for the identity key is preserved; the only new primitive is an ECDSA-P-256 signature by the StrongBox key, in its natural role (hardware attestation key).

**Bad / accepted limits (disclosed):**
- **Curve25519 private keys are TEE-resident, not StrongBox-resident** (D7) — bound to StrongBox, not inside it. Accepted under the declared threat model and disclosed at pairing, consistent with ADR-029's TEE posture.
- **One extra wire field + one extra verification step** (`child_binding_sig`, check 4b) — a `proto` change carried by the implementing issues; modestly more parent-side surface.
- **`K_bind` is a third per-device key** to generate and retain; small additional StrongBox slot + keygen latency at pairing.
- **This ADR ships no code.** It is a canon-correction + design-ratification record (cf. ADR-025/031). Implementation lands in #22 (child) and the ADR-025 D5 parent-side issues, each with the tests below, after the D-context bench confirmation.

## Blocking before implementation (binds #22 + the parent-side issues, not this ratification)

- **Bench confirmation (D-context gate).** A maintainer confirms the StrongBox/Curve25519 behavior on a real Pixel 7 before code lands. If Curve25519 *is* StrongBox-attestable, withdraw this ADR. The bench check MUST also confirm the **Tier-2 TEE** behavior of `K_bind` (P-256 with `setIsStrongBoxBacked(false)` → `TRUSTED_ENVIRONMENT` attestation) to validate the Tier-2 mapping (D1).
- **`child_binding_sig` encoding (signer/verifier drift guard).** Android `Signature.sign()` returns **DER-encoded** ECDSA (variable length). The wire encoding is `hex(DER)` (the `SpkiAssertion`-hex precedent, ADR-031); the verifier MUST parse DER, and `pair-06`'s vector MUST fix the exact byte encoding. (The protocol now mixes `base64url` Ed25519 sigs with `hex` ECDSA — pin this down in the implementing issue.)
- **Wire the signer to `K_id`, not `K_bind`.** `IdentityKeyProvider.sign()` MUST use the **Ed25519 `K_id`** (for `SpkiAssertion`, ADR-031), never the P-256 `K_bind` — an easy mis-wire since both are generated together at pairing. Add an explicit test asserting the assertion verifies under the Ed25519 identity key.
- **Inherited ADR-025 blockers (still open, not discharged here).** ADR-025's "pairing-endpoint pre-auth / rate-limiting" item is inherited unchanged (and ECDSA verify + an extra keygen give a hostile pre-pin endpoint marginally more compute to burn). The "abort/cleanup fail-closed" item is satisfied by D7. **Full-transcript replay binding:** `ChildKeyBinding` binds `{K_bind (via check 4), child_ed25519_pub, child_x25519_pub, this-attempt nonce}` and the parent keys ride in via the SAS — this binds *most* of the transcript and gives per-attempt freshness, but ADR-032 does **not** claim to fully discharge ADR-025's transcript-binding item; the parent-side issue MUST confirm a spliced/replayed POST cannot pin (the single-use nonce + SAS + check 4b are the current defense).
- **`K_bind` lifecycle.** Define `K_bind` alias, that its private key never leaves StrongBox, that it is generated once at pairing and reused for periodic re-attestation (D6), and that loss/rotation of `K_bind` forces re-pair (it is per-device, non-recoverable, like the identity keys).
- **Binding-sig determinism.** ECDSA-P-256 is non-deterministic by default; the verifier MUST NOT depend on signature determinism. Canonicalize the signed body via JCS (D2) identically on both sides; the signer signs `JCS{v,child_ed25519_pub,child_x25519_pub,provisioning_nonce}`, the verifier re-canonicalizes the received fields. Reject-on-throw for any parse/decode failure (mirror `/lock` body-parse).
- **Byte-level pubkey validation (ADR-025 D6 still applies).** `child_ed25519_pub`/`child_x25519_pub` MUST base64url-decode to exactly 32 bytes with a valid curve encoding; the P-256 leaf key is validated as part of cert-chain parsing.
- **Fail-closed signer.** Until `K_bind` + `K_id` exist (pre-pairing), `IdentityKeyProvider` returns `null` and `SpkiBindingSigner` produces no assertion (ADR-031 D5) — unchanged.
- **Key-use authentication policy (decide at implementation).** The prior CRYPTO §3 sample set `setUserAuthenticationRequired(true)` + biometric/credential params on the identity key; the corrected sample omits them because `K_id` signs the runtime `SpkiAssertion` (ADR-031) **server-side with no human present**, so a per-use biometric gate would break automated signing. The implementing issue MUST decide explicitly: `K_id`/`K_enc` keep `setUnlockedDeviceRequired(true)` (device must be unlocked) but **not** per-use `setUserAuthenticationRequired` (which would deadlock runtime signing); `K_bind` (used only at pairing/periodic-attestation, where the device is being operated) MAY require user auth. This is a behavior decision surfaced for sign-off, not silently dropped.

## Test plan (binds the implementation issues, not this ADR)

- **Binding-sig accept:** a `child_binding_sig` produced by a P-256 key over `JCS{v:1, ed, x, nonce}` verifies against that same (attested-leaf) key with matching fields ⇒ proceed.
- **Binding-sig reject (each, fail-closed, deterministic / seam-injected):** wrong signing key (attacker's P-256 key, not the attested leaf); body field substituted (`child_ed25519_pub` **or** `child_x25519_pub` swapped) so the signature no longer covers the presented keys; stale `provisioning_nonce` in the body; malformed/empty `child_binding_sig`; leaf key ≠ presented `K_bind`.
- **Attestation verify (amended §7.3, P-256 leaf):** good Google-rooted STRONGBOX chain over `K_bind` with matching nonce + GREEN + locked + allow-listed model + valid binding sig ⇒ proceed; each of {wrong root, nonce mismatch, `verifiedBootState != VERIFIED`, unlocked bootloader, non-allow-listed model, `securityLevel != STRONGBOX`, leaf ≠ `K_bind`, binding-sig fail} ⇒ refuse pair.
- **Four-key SAS (unchanged regression, ADR-025 D2a):** substitution of any of the four pinned keys changes the SAS.
- **Test vectors** under `docs/test-vectors/pairing/`: `pair-06-binding-ok` (a real Pixel-7 P-256 STRONGBOX attestation chain + valid `ChildKeyBinding`/`child_binding_sig`, accept), `pair-07-fail-binding-sig` (binding-sig over substituted keys / by the wrong key), and `pair-08-fail-binding-stale-nonce`. Names start at **06** to avoid colliding with the existing `pair-02-fail-not-verified` (§9.1), ADR-025 D6's `pair-03-fail-malformed-pubkey`, and **ADR-029's `pair-04-tier2-tee-ok` / `pair-05-fail-unknown-oem-root`**. A Tier-2 TEE binding vector rides on ADR-029's `pair-04`.
