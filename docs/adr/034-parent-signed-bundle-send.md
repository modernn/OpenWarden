# ADR-034: Parent assembles, signs, and sends policy bundles (monotonic policy_seq) — issue #27

Status: Proposed
Date: 2026-06-23
Relates: **ADR-033** (the parent root key this signs with — resolves its deferred "#27 signing key" sign-off item), **ADR-015/019** (the one signing rule — Ed25519 over RFC 8785 JCS minus `sig`), **ADR-017** (replay/rollback: `policy_seq` monotonic floor + freshness window), PROTOCOL.md §2 (PolicyBundle field set) / §3 (child `verify_bundle`) / §5 (replay + freshness); docs/ATTACKS.md (H1/C8 replay = critical), docs/DEFENSES.md

## Context

Issue #27 (`area:parent-kmp`, `priority:high`, `crypto`, `agent-blocked`): the parent assembles a `PolicyBundle`, signs it (Ed25519 + JCS, monotonic `policy_seq`), and POSTs it to the child `/policy`. This is the parent→child **control path** — what makes the parent app actually push policy. #24 (ADR-033, merged) provides the root key to sign with; this is its first consumer.

**The wire format + child verification already exist** — nothing in `proto` changes:
- `proto/Bundle.kt` defines `PolicyBundle`/`Policy` (PROTOCOL §2 field set, snake_case wire names).
- `PolicySigner.signingBytes(bundle)` is the canonical signing input (JCS minus `sig`), already pure + tested.
- The child `/policy` endpoint (`ApiServer` → `PolicyAdmission.admit`) runs the full ADR-017 admission: JCS bounds, audience (`child_device_id`), signature, replay floor (`ReplayFloor`: `MAX_SEQ_JUMP=1024`, `GENESIS_FIRST_VALID_SEQ=1`), freshness window, two-phase durable apply (floor advances LAST).

**This issue is purely parent-side.** What does not exist yet: a parent monotonic `policy_seq` sequencer, bundle assembly, the signing wire-up, and a `/policy` POST.

The maintainer-approved forks (front-loaded): sign via **BC `RootKeyProvider.sign`** (not libsodium); **root key signs directly, foreground push** (the ADR-033-deferred decision); **24h configurable freshness window**.

## Decision

**D1 — Bundle assembly (`PolicyBundleBuilder`, pure/`commonMain`).** Build a `PolicyBundle` per PROTOCOL §2/§3: `v=1`, `policy_seq` (D3), `child_device_id` (D4), `issued_at = now`, `not_before = now`, `not_after = now + freshnessWindow`, `nonce` = CSPRNG 16 bytes hex (32 chars, fresh per bundle), `policy` (from `PolicyEditor.toProtoPolicy()`). `now`/`nonce` are injected (a `() -> Long` clock + a `NonceGenerator` seam) so assembly is deterministic in tests.

**D2 — Signing: the root key signs directly, via Bouncy Castle, foreground.** The signature is `RootKeyProvider.sign(PolicySigner.signingBytes(unsigned))` hex-encoded into `bundle.sig` (`SignedBundleAssembler`, `commonMain`). This reuses ADR-033's BC Ed25519 path (host-testable, RFC 8032, proven interoperable with the child's libsodium verifier by ADR-033's RFC KATs) and **sidesteps the unverified ionspin `seedKeypair` API** that the libsodium `PolicySigner.sign` path would need. **The auth-gated root key signs directly (v1):** the parent pushes policy in the **foreground**, just-authenticated (the StrongBox MasterKey's `setUserAuthenticationRequired(true)` means a fresh unlock gates each push — acceptable as "biometric-to-push"). This **resolves ADR-033's deferred #27 decision**: a derived, non-auth-gated **session key for background push is deferred** (it needs the child to pin it or accept a root-signed delegation — a protocol addition beyond #27). **Fail-closed: if the provider is not provisioned (`sign` returns `null`), no bundle is produced and nothing is sent** — never an unsigned bundle.

**D3 — Parent monotonic `policy_seq` (`PolicySeqStore` seam + durable Android impl).** The parent is the authoritative monotonicity anchor (PROTOCOL §5). `reserveNext()` durably persists and returns a **strictly-greater** seq, starting at `GENESIS_FIRST_VALID_SEQ = 1`; it **never returns a value ≤ one already issued** (monotonic, never reused). The Android impl mirrors the child `ReplayFloorStore`: `EncryptedSharedPreferences` under a StrongBox MasterKey, `commit()` + **readback** check — fail-closed (throws) on a failed durable write so the parent never issues a seq it didn't persist. A reserved seq **binds to one immutable signed bundle**: a transient send failure retries the *same* bundle (same seq, idempotent at the child); only a new/changed policy reserves a new higher seq. Gaps are fine (the child accepts any `seq > floor` within `MAX_SEQ_JUMP`); seq reuse for *different content* is the thing this forbids.

**D4 — `child_device_id` from a `PairedChildStore` seam (fail-closed if unpaired).** The audience binding is the pinned child Ed25519 pubkey (base64url, 32 bytes), established at pairing (#23, not yet built). `PairedChildStore.pairedChildId()` returns it or `null`; **the sender refuses to send (fail-closed) when no child is paired.** This decouples #27 from #23 — pairing drops the value in later; tests inject it.

**D5 — Transport (`PolicyTransport` seam; `ChildApiClient.postPolicy` impl).** `PolicySender` serializes the signed bundle to JSON (the §2 wire shape) and hands it to `PolicyTransport.postPolicy(json)`, which POSTs to the child `/policy`. The Android impl extends the existing demo `ChildApiClient` (ktor/OkHttp, `ApiResult`), parsing the child's `200 {"status":"applied","policy_seq"}` / `400 {"error","reason"}`. Transport stays demo-grade (hardcoded URL) until real mDNS + pinned TLS (#21/#23) — confidentiality is orthogonal to this signed-bundle authenticity (ADR-030/031).

**D6 — Fail-closed orchestration (`PolicySender`, `commonMain`).** Order: (1) require provisioned root key **and** a paired child — else return a typed failure, **consuming no seq**; (2) `reserveNext()`; (3) build; (4) sign (else fail-closed); (5) `postPolicy`. A transport failure returns the signed bundle for an idempotent retry. `PolicySender` never emits an unsigned or unaddressed bundle and never lowers/reuses a seq.

**D7 — Tests, host-side.** `commonTest`/`jvmTest`: `PolicyBundleBuilder` field/timestamp/nonce contract; `PolicySeqStore` double monotonic + never-reuse; `PolicySender` orchestration (advance only past the pre-checks, fail-closed on no-key / no-pair / transport-fail) with fake provider/transport. `androidUnitTest`: `SignedBundleAssembler` real **BC sign → verify round-trip** + **tamper-invalidates-signature** (every mutated field breaks verify), via `StoredRootKeyProvider` + an in-memory storage double provisioned with derived keys.

## Consequences

- **The child does NOT yet enforce the freshness window (v1) — this ADR makes no end-to-end-freshness claim.** The parent signs `not_before`/`not_after`, but the child `PolicyAdmission` does not yet evaluate them (no `EXPIRED`/`CLOCK_SKEW` step — PROTOCOL §2.1 steps 9–10; the child code carries a TODO admitting this). So the 24h window is **inert end-to-end until child-side enforcement lands** (tracked follow-up, `area:child-android`). The parent signs the window correctly, so it activates the moment the child checks it. v1 therefore ships the parent half of the replay-window defense with the child half tracked — a maintainer-accepted residual, not a closed loop.
- **The send library has no foreground caller yet.** #27 ships `PolicySender` as the library; the parent UI that wires a fresh device unlock → push (the "biometric-to-push" property D2 leans on, and ADR-033's sign-off condition) is a tracked follow-up. The auth-gating guarantee holds **only** once that wiring exists; until then `PolicySender` MUST be invoked solely from a just-authenticated foreground context (a background/cached-unlock caller would silently void the guarantee).
- **The child verifies by parse-then-re-canonicalize (ADR-019 tension).** The child decodes the bundle and re-canonicalizes its typed object to verify, rather than verifying over the received bytes (PROTOCOL §3.1 / ADR-019). This is **correct today** — the parent's `signingBytes` and the child's canonicalizer are pinned byte-identical by the golden-vector twin tests (`proto` `CanonicalTest` ↔ child `PolicyAdmissionTest`) plus a real libsodium-signature-verifies-at-child KAT — but it is a canon-vs-code tension to reconcile (ratify the parse-and-re-canonicalize exception + extend the golden vector to a unicode bundle, or move the child to verify over received bytes via the existing `SigningInput.forDocument`). Pre-existing, child-side; surfaced here.
- **The `seq → bundle` binding is in-process.** A reserved seq binds to its signed bundle within one process; a crash between `reserveNext()` and send loses the binding (the next `send()` produces a different bundle under a higher seq — a gap the child tolerates within `MAX_SEQ_JUMP`). **No seq is ever reused for different content**; the crash case only burns a seq.
- **Full parent-anchor reconciliation deferred.** PROTOCOL §5's "parent reads the child's reported floor/chain tip and pushes above the highest ever seen on regression/fork" needs the child to report its floor (sync/ack plumbing) — #27 ships the monotonic local sequencer + ack parse; the regression/fork re-anchor is a follow-up.
- **Transport is still demo-grade** (hardcoded cleartext `http://10.0.2.2:7180`); authenticity (this ADR) holds over any transport (the bundle carries no content), confidentiality lands with #21/#23 (ADR-030/031).
- No `proto` change; no new crypto primitive (reuses ADR-033 BC Ed25519 + ADR-015 JCS).
- On merge this ADR flips to **Accepted**.
