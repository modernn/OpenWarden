# OpenWarden Internationalization (i18n) Strategy

**Status:** v1 ships English only (en-US). This document defines the discipline now so future translation is mechanical, contributor-friendly, and culturally aware.

Cross-references: [DESIGN_PARADIGMS.md](./DESIGN_PARADIGMS.md), [ONBOARDING.md](./ONBOARDING.md), [SIMPLIFY.md](./SIMPLIFY.md), [UX_PATTERNS.md](./UX_PATTERNS.md), [KID_TRANSPARENCY.md](./KID_TRANSPARENCY.md).

---

## 1. Strings discipline

Every user-facing string lives in a resource bundle. No exceptions.

- **Android:** `app/src/main/res/values/strings.xml`
- **iOS:** `Localizable.strings` (and `Localizable.stringsdict` for plurals)
- **Shared KMP modules:** common resource bundle accessed via `moko-resources` or equivalent, surfaced to both platforms

Rules:
- **No hardcoded strings in code.** Not in Composables, not in SwiftUI Views, not in ViewModels, not in error messages, not in logs that ever surface to a user.
- **Descriptive keys.** Keys describe *what the string is for*, never the English content. Good: `kid_blocked_reason_bedtime`, `parent_unlock_now_button`, `onboarding_step_2_title`. Bad: `text1`, `the_quick_brown_fox`, `click_here`.
- **Context comments are mandatory.** Every key has a comment explaining where it appears, who reads it (kid vs. parent), and any constraints (max length, tone).
- **ktlint custom rule** fails any PR introducing a hardcoded string literal inside a `Text(...)` Composable or `Toast.makeText`. SwiftLint mirror rule on the iOS side flags `Text("...")` literals.

## 2. Translation workflow

- **Source language:** en-US. All keys authored in en-US first.
- **Translation platform:** [Weblate](https://weblate.org), self-hosted at `weblate.openwarden.dev` or piggybacked on `hosted.weblate.org` (free for OSS).
- **Why Weblate:** OSS, contributor-friendly web UI, glossary support, translation memory, screenshot context, and direct Git integration. Crowdin remains a paid fallback if we outgrow Weblate; we defer that decision.
- **Contribution flow:** translators sign in to Weblate, see strings with context comments and (where useful) screenshots, submit translations. Weblate opens a PR to `main`.
- **Maintainer review:** a weekly cadence. Maintainer merges Weblate PRs after a sanity scan. No merge without a native-speaker review for kid-facing strings.

---

## 3. Target languages

Demand-driven. We do not pre-localize speculatively.

- **v2 targets:** en-US, es-ES, es-MX, pt-BR, fr-FR, de-DE
- **v3 expansion:** ja, ko, zh-CN, ar (our first RTL test case), hi, ru
- **Beyond v3:** ship what users ask for. A translation contributor showing up with a complete first pass is the strongest possible signal of demand.

---

## 4. RTL support

RTL is treated as a first-class layout direction from day one, even while we ship en-US only.

- **Compose:** wrap previews and instrumentation tests with `CompositionLocalProvider(LocalLayoutDirection provides LayoutDirection.Rtl)`. Every screen has an RTL preview.
- **SwiftUI:** uses native `.environment(\.layoutDirection, .rightToLeft)`.
- **Shared:** a small `LocaleManager` abstracts current locale + direction across platforms; UI consumes the abstraction, not platform APIs directly.
- **Tests:** instrumentation tests run Arabic + Hebrew locales as a smoke screen, even before we ship translations. Layout regressions caught now are free; caught after Arabic ships, expensive.

---

## 5. Kid-readable copy

OpenWarden copy targets kids. Translation is not enough; tone matters.

- All kid-facing copy targets a **9-12 reading level**.
- Verified with Common Sense Media reading-grade tools and Flesch-Kincaid checks in CI on en-US source strings.
- **Pictograms first** (see [KID_TRANSPARENCY.md](./KID_TRANSPARENCY.md)). Text is supplementary. An icon translates universally; a sentence does not.
- **Native-speaker review required** for kid-tone in any target language. A dictionary-correct translation that sounds clinical or condescending to a 10-year-old is a defect. Reviewers explicitly check: "would a Spanish-speaking 10-year-old read this without rolling their eyes?"

## 6. Contributor workflow

The contract for adding strings:

1. Developer adds a new key to `strings.xml` (and shared resource bundle) with the English text and a context comment.
2. Weblate auto-picks up the new key on next sync. Translators get notified for any language flagged as "supported."
3. CI fails any PR that ships a new key without an English value or context comment.
4. **New language:** create `values-XX/strings.xml`, add the language to Weblate's project config, and Weblate handles the rest.
5. **Translation completeness gate:** CI fails if a language declared "supported" (in `i18n.toml`) has untranslated keys. Languages in "community preview" don't gate CI.

## 7. Pluralization

- **Android:** `<plurals>` with ICU MessageFormat for complex cases.
- **iOS:** `Localizable.stringsdict`.
- **Both:** strictly follow CLDR plural rules (one / two / few / many / other). English-brain shortcuts like `if (n == 1) "1 minute" else "$n minutes"` are a bug; Polish, Arabic, and Russian have multiple plural forms English does not.
- Time-related strings ("3 minutes left", "1 hour remaining") are the most common offenders. Every such string must use the plural API.

---

## 8. Date/time formatting

- `kotlinx.datetime` with locale-aware formatters on the Kotlin side; `DateFormatter` with `locale = .current` on iOS.
- **Never hardcode** patterns like `"yyyy-MM-dd"` or `"h:mm a"` in user-facing strings. Use locale-default skeleton patterns (`DateTimeComponents.Format`, `DateFormatter.dateStyle`).
- Internal logs and protocol payloads continue to use ISO 8601; that is not user-facing.

---

## 9. Number formatting

- `NumberFormat.getInstance(locale)` (Android/Kotlin), `NumberFormatter` (iOS).
- Decimal separators, digit grouping, and digit shaping (e.g. Arabic-Indic digits) follow the user's locale.
- **No currency formatting.** OpenWarden does not bill, does not show prices, does not handle money. The currency API surface is intentionally absent.

---

## 10. Right-to-left layout

- All padding/margin uses start/end, never left/right.
- Icons that imply direction (chevrons, arrows, swipe affordances) mirror automatically via `autoMirrored = true` (Compose) or `imageFlipsForRightToLeftLayoutDirection` (UIKit underlying SwiftUI).
- **Avoid directional metaphors in copy.** "Swipe right to approve" becomes "Swipe to approve" with an icon that mirrors. "Continue ->" becomes a button with an auto-mirroring chevron.
- Arabic emulator/simulator is in the manual QA checklist for any UI-heavy PR.

---

## 11. Audio/voice

- **TTS for non-readers:** optional, locale-aware. Uses platform TTS (Android `TextToSpeech`, iOS `AVSpeechSynthesizer`), which speaks current-locale strings natively. No translation pipeline needed; the OS handles it.
- **No shipped voice clips.** Pre-recorded audio would multiply asset size per language and require a voice contributor per locale. TTS scales for free.

---

## 12. Cultural sensitivity

Parental controls are not culturally neutral. We design for that:

- Translation reviewers explicitly flag culture-specific concerns in PR comments.
- **Known landmines:**
  - **Calendar:** religious holidays affect "school day" defaults. Eid, Ramadan, Diwali, Lunar New Year all impact what a "normal week" looks like.
  - **School-week vs work-week:** Friday-Saturday weekends in Saudi Arabia, UAE (until recently), Iran. Our "weekday" / "weekend" rules must be locale-driven, not US-default.
  - **Bedtime norms:** dinner and bedtime vary widely (Spain runs late). Defaults shipped per-locale, not globally.
- **v1/v2 stance:** ship US/EU defaults; let parents override. Locale-specific defaults are a v3 problem, and we defer rather than guess.

---

## 13. F-Droid metadata localization

The store listing itself is translatable:

- `metadata/en-US/short_description.txt`
- `metadata/en-US/full_description.txt`
- `metadata/en-US/images/phoneScreenshots/`
- Per-locale folders (`metadata/es/`, `metadata/ar/`, ...) hold translated descriptions and locale-specific screenshots.
- Weblate manages the text components alongside app strings; screenshots are contributor-submitted PRs.
